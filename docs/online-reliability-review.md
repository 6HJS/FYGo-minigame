# 在线对局可靠性检查（2026-03-30）

## 总体结论
目前联机链路是“HTTP 建房/进房 + WebSocket 同步”的混合模型，方向是对的，但实现层存在多处**时序竞争、断线恢复不完整、协议缺少确认**的问题，容易出现你描述的“不可靠”。

---

## 主要风险点（按优先级）

### P0：WebSocket 关闭时 Promise 清理顺序错误，导致上层无法感知失败
- 客户端 `connectWebSocket()` 在 `onClose` 回调中先把 `socketOpenReject` 置空，再尝试调用它。
- 结果：如果连接在握手阶段失败，调用方可能收不到 reject，进入“看起来在等、但其实永远不会成功”的状态。
- 影响：建房/进房后 `ensureRoomSubscribed()` 等流程会偶发卡死或误判。

**建议**
1. 在清空引用前保存 `const reject = this.socketOpenReject` 并执行。
2. 统一在一个 `finalizeSocketOpen(err?)` 函数里处理 resolve/reject，避免分支重复导致遗漏。

### P0：缺少自动重连策略（只有心跳，无恢复）
- 客户端心跳只做 `ping`，但 `onClose` 后没有重连尝试。
- 服务端每 5 秒心跳剔除“无 pong 连接”。移动网络抖动时极易被踢下线。
- 影响：用户前台弱网切换、短暂后台恢复后，房间状态不会自动回补。

**建议**
1. 增加指数退避重连（例如 0.5s / 1s / 2s / 4s，最大 10s）。
2. 重连成功后自动：`subscribe_room` -> `fetchRoomState` 对账。
3. 达到最大重试次数后再提示用户“请手动重连”。

### P1：动作发送无 ACK / 重放机制，丢包后 UI 与服务端可能分叉
- `sendAction()` 目前只 `send()` 后立即返回 true，没有“服务端已接收”的确认。
- 服务端广播虽然带 `snapshotVersion`，但客户端没有做“请求-应答关联”。
- 影响：弱网下 action 丢失时，前端可能误以为落子成功。

**建议**
1. action 增加 `clientActionId`。
2. 服务端返回 `action_ack{clientActionId, snapshotVersion}`。
3. 客户端超时未 ACK 自动重发（幂等处理由服务端保证）。

### P1：快速匹配逻辑与“排队”语义不一致
- `enqueueQuickMatch()` 在未匹配到对手时，会先把自己放队列，再立即 `createRoom()`。
- 这会形成“队列 + 独立房间”双轨状态，恢复和清理路径复杂。
- 影响：高并发下容易出现匹配状态歧义、房间残留和重复占位。

**建议**
1. 二选一：
   - 要么纯队列，匹配成功再创建房间；
   - 要么纯房间池，不维护独立 queue。
2. 给匹配请求引入 `matchTicketId`，并在所有状态流转中携带。

### P1：鉴权模型偏弱（playerToken 为客户端长期可复用令牌）
- 当前 `playerId/playerToken` 由客户端本地生成并长期保存。
- 这更像“设备标识”而非服务端签发凭据。
- 影响：被抓包/复制后可在其他端伪装重连或顶号。

**建议**
1. 改为服务端签发短期会话票据（JWT 或随机 session token）。
2. `join/create` 后返回 `wsSessionToken`，WebSocket 连接仅接受短期 token。
3. 增加 token 轮换与撤销。

### P2：订阅确认策略固定 5 秒 + 12 次，复杂网络下容易误报失败
- 目前是固定 350ms 重试，共 12 次（约 4.2 秒）+ 外层 5 秒超时。
- 影响：在跨地域、高抖动链路上容易“其实成功但被判失败”。

**建议**
1. 改指数退避 + 抖动，避免节奏碰撞。
2. 超时时长由网络质量动态调整（或可配置）。

### P2：观测性有日志，但缺少关键可靠性指标
- 服务端日志较多，但没有结构化指标汇总。
- 影响：线上难以快速定位“不可靠”到底是断连、鉴权、订阅还是 action 丢失。

**建议**
1. 增加指标：
   - ws 连接成功率
   - 平均重连次数
   - 订阅确认耗时 P50/P95
   - action ACK 超时率
2. 每个 room/player 打 `traceId`，串联 HTTP + WS 全链路。

---

## 建议落地顺序（两周版）

### 第 1 周（止血）
1. 修复 `onClose` 的 Promise reject 顺序问题（P0）。
2. 加最小可用自动重连（3~5 次指数退避）。
3. 重连后强制 `fetchRoomState` 对账。

### 第 2 周（增强）
1. 引入 `clientActionId + action_ack`。
2. 统一快速匹配状态机（移除 queue/room 双轨）。
3. 引入服务端短期会话 token。

---

## 回归测试建议（必须自动化）
1. **断网 3 秒恢复**：对局中拔网线/开飞行模式后恢复，双方盘面最终一致。
2. **双端同账号抢占**：新连接上线后旧连接应被稳定踢下，且房间状态不抖动。
3. **消息乱序与重复**：注入延迟/重复包，验证 action 幂等与最终一致性。
4. **高延迟 300~800ms**：订阅确认不能频繁误报失败。
5. **服务端重启**：客户端应自动重连并恢复到最新快照。

---

## 你现在就能做的最小改动
如果只做 3 件事，优先：
1. 修复握手失败 reject 不触发。
2. 做自动重连 + 重连后拉取房间快照。
3. 给 action 增加 ACK 与超时重试。

这三件会显著降低“偶发失联、操作无响应、双方不同步”的投诉概率。
